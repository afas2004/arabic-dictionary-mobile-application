Arabic Dictionary — UI Round 2

Updated: highlighting computed by one JS function (no per-card mistakes), original Flutter UI restored (COMMON / Weak Root / from-baseform / romanisation), original 4×3 conjugation grid, missing screens included.

What changed since the last pass:
  • Highlighting is now algorithmic — every card runs through highlightExact() or highlightRoot() at render time. No hand-coded spans means no duplicate or missing highlights. Rule: exact highlights the matched substring, root highlights each of the three root radicals wherever they appear.
  • Original UI elements restored: COMMON badge (green, left side), Weak Root badge (red), "from كَتَبَ" base-form display for derived verbs, romanisation in parentheses, verb-form label (Form I / II / …).
  • Conjugation table uses the original Flutter grid: 4 columns (أنتِ · أنتَ · هي · هو), 3 rows (Singular · Dual · Plural), plus a bottom strip for نحن (We) / أنا (I). Imperative block is the simpler Singular / Dual / Plural layout.
  • Missing screens added: empty state, recent searches, favourites, no-results, settings with a theme-colour picker placeholder.
  • Only semantic colour is Material Blue #1976D2 (matches Color(0xFF1976D2) already in your codebase). Red is reserved for Weak Root / mutation letters as you already use it; green is reserved for COMMON as you already use it.

6. Ranking logic (list page)

For a query Q, the cascade is run once for exact, then once after clitic stripping, and the root of Q is used to fetch the full family. Results are unioned by lexicon.id and sorted:

RankMatch typeSourceExample (Q = كتب)
1Exact lemma match (normalised)lexicon.form_stripped = Qكَتَبَ, كُتُب
2Exact conjugation hit (if lemma not matched)conjugations.form_stripped = Qfor Q = يكتبون → كَتَبَ
3Exact after clitic stripping (T3–T6)stemmer → lexiconfor Q = بالمدرسة → مَدْرَسَة
4Same root familylexicon.root = extracted rootكِتَاب, مَكْتَبَة, كَاتِب…
5Prefix match on lemmaform_stripped LIKE 'Q%'Q = كت → كَتَف
6Substring matchform_stripped LIKE '%Q%'rare on Arabic side, common for English
7Fuzzy / pattern-based rootKhoja pattern → rootonly fires when ranks 1–6 return nothing

Within each rank: is_common DESC, frequency DESC, length ASC.

Root-family cap: start at 10 cards, then fall through to prefix matches with a "…N more in this root" expand row. Keeps common roots (كتب, قول, فعل) from drowning prefix hits. Tune after real use.
Highlight policy — exact or root only: every card either highlights the contiguous substring of Q inside the lemma (ranks 1–3, 5, 6) or highlights each of the 3 root radicals individually (rank 4 and when ranks 1–3 share a root with Q). Ranks 5 and 6 still work because prefix/substring implies contiguous presence. If neither applies, the card renders unhighlighted — never a duplicate or missing radical, guaranteed by the shared JS function at render time.

8. Decisions still worth flagging

a. Theme-colour personalisation — done as a placeholder

Alongside this HTML there's now a ThemeController and a SettingsScreen in your Flutter project. All accent colour references going forward should use Theme.of(context).colorScheme.primary instead of the hardcoded Color(0xFF1976D2) — this means when you eventually wire the settings swatches into the controller (currently hardcoded to blue), the whole app recolours without more refactoring.

b. Highlighting: EXACT or ROOT, never both

For rank 1 (exact lemma match), you could argue both the exact span AND the root radicals should be highlighted. I've kept it to exact only for rank 1 on purpose — otherwise the highlight reads as noise rather than signal. Root highlighting only kicks in when the query doesn't appear literally in the lemma.

c. Cap root-family at 10, then let prefix take over

Tuned later once you see real traffic. For the viva dataset 10 is plenty — no root in the FYP test set has more than 12 entries.

d. Still to build (not mocked)

  • Text-size picker (settings row is there as a stub — needs a range slider screen)
  • Share-sheet system integration (just calls platform share)
  • L2 disk cache read/write paths (parked from the caching doc)
  • Word-model MatchProvenance field (needed to drive the highlight mode decision without re-running the stemmer at render time)
